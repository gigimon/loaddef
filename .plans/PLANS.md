# Description
Цель: сделать небольшой Rust benchmark-сервер для воспроизводимого сравнения нагрузочных фреймворков (включая vikhry) по пропускной способности, латентности и стабильности под ошибками/медленными ответами/крупными payload.

Базовые требования:
- Endpoint `200 OK`
- Endpoint `404/500` (управляемая доля ошибок)
- Медленный endpoint с рандомной задержкой `100..1000 ms`
- Endpoint с рандомным payload от сотен байт до десятков килобайт
- Прозрачная статистика по количеству запросов и latency
- Встроенный веб-интерфейс за время текущего запуска: графики, total requests, суммарный RPS за период, разбивка запросов по endpoint.

Принятые решения:
- Ошибки (`404/500`) генерируются полностью случайно (без фиксированного seed), чтобы имитировать «живую» среду.
- Основной канал наблюдения: встроенный Web UI с накоплением метрик в памяти процесса за время текущего запуска.
- Таймсерии хранятся полностью за весь период работы процесса (без ring buffer).

Технологии (предлагаемый стек):
- ЯП: Rust (stable)
- Runtime: `tokio`
- Web framework: `axum`
- Метрики: in-memory counters + time series (опционально позже Prometheus endpoint)
- Латентность/гистограммы: `hdrhistogram`
- Сериализация: `serde`, `serde_json`
- RNG: `rand`
- UI: встроенная web-страница dashboard (подача как static assets из backend).

## Step 3: Нагрузочные endpoint
[x] `GET /ok` -> всегда `200`, небольшой JSON.
[x] `GET /e404` -> всегда `404`.
[x] `GET /e500` -> всегда `500`.
[x] `GET /slow` -> потоковая отдача chunk-ов с паузами `100..1000 ms` между chunk-ами (выбран режим стриминга).
[x] `GET /blob?min=...&max=...` -> случайный payload в диапазоне размеров (напр. 256B..64KB), генерируемый заново на каждый запрос.
[x] Поддержать режим без seed (основной) и опциональный seed для отладки.

## Step 4: Сбор статистики и Web UI
[x] Глобальные атомики: total requests, по endpoint, по status code.
[x] Латентность: histogram по endpoint (минимум p50/p95/p99).
[x] Таймсерии в памяти по секундам: requests/sec (и по endpoint), errors/sec, bytes/sec.
[x] API для UI: `GET /api/stats/summary`, `GET /api/stats/timeseries`.
[x] Dashboard `GET /` с графиками: RPS за период, total requests, средний RPS за весь период, breakdown по endpoint.
[x] Обновление UI через polling (1 раз в секунду) к `/api/stats/*`.
[] Опционально: `GET /metrics` (Prometheus) для внешней верификации.
[x] `POST /api/stats/reset` -> сброс статистики между прогонами.

# Решенные развилки
- Модель ошибок: `B` (полностью случайная, без fixed seed).
- Медленный endpoint: `A` (streaming chunks с задержкой между chunk-ами).
- Большой payload: `A` (генерация случайных байт на каждый запрос).
- Канал статистики: `C` (встроенный Web UI + in-memory данные за текущий запуск).
- Хранение истории: `A` (полная история с момента старта процесса).
- Обновление dashboard: `A` (polling раз в 1 секунду).
- Ошибочные endpoint: `A` (раздельные `/e404` и `/e500`).
